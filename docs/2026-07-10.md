## 4. Issues Caught by Review (before landing)

Each of the following was found by an independent reviewer subagent — not self-reported by the implementer — and fixed before the task was allowed to proceed:

1. **Test A2:** `test_risk_explainer.py`'s original fixture only ever triggered regex-sourced red flags, so the assertion loop never proved that a keyword-DB-sourced finding's real CSV-authored `reason` wins over the generic fallback string. Fixed by adding a fixture known to trigger a `keyword_db` finding and asserting the precedence directly.
2. **Test B1:** `test_async_api.py` still tested `status="running"` after the rename — it passed, but only because `"running"` fell through to an unrelated code path, not because it exercised the real in-flight value. Fixed to test `"processing"`, the value production code now actually emits.
3. **Bug B2:** `retry_analysis()`'s original implementation checked status/retry_count with a `SELECT`, then performed an unconditional `UPDATE` — a genuine check-then-act race. Two concurrent retry calls (e.g. a double-clicked Retry button) could both pass the guard, pushing `retry_count` past its cap and double-submitting the same analysis to the worker. Fixed by making the `UPDATE` itself the atomic guard (`WHERE status IN (...) AND retry_count < _MAX_RETRIES`, branching on `cursor.rowcount`).

One additional gap was found and closed without a review cycle: the B2 implementer discovered `database.get_result()`'s `SELECT` didn't include `retry_count` at all — the plan's own test would have raised `KeyError` without this fix. Confirmed necessary by the reviewer, not scope creep.
