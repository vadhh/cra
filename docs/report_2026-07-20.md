## 3. Summary of Changes Today (2026-07-20)

### 3a. Fix: `datetime.utcnow()` deprecation (punch-list item 4)
* Replaced all 3 call sites — [database.py](file:///home/stardhoom/LDV/ldv-backend/database.py) (`create_document`, `purge_expired`, `write_audit`), [manage.py](file:///home/stardhoom/LDV/ldv-backend/manage.py) (`backup_cmd`), and `tests/test_sprint4.py` — with `datetime.now(timezone.utc)`.
* `database.py`/`manage.py` call sites needed `.replace(tzinfo=None)` to preserve the existing naive-UTC string comparisons (`expires_at < ?` and backup-folder-name pruning) — a plain swap would have raised `TypeError: can't compare offset-naive and offset-aware datetimes` on the backup-pruning path.
* Full suite: 107/107 passed, warnings down from 26 to 3 (remaining 3 are pre-existing, harmless, third-party SWIG/tokenizer warnings, unchanged from 07-17 §4a).

### 3b. New: Original-11 corpus validation harness (punch-list item 2, in progress)
* Added [run_original11_corpus.py](file:///home/stardhoom/LDV/ldv-backend/tests/run_original11_corpus.py) — a standalone runner (kept separate from `run_benchmark.py`, whose schema only covers language/jurisdiction/clause-presence) that executes real fixtures end-to-end through L1–L3 and PDF export, for the 11 original profiles specifically, and reports expected-vs-actual instead of pass/fail-on-crash.
* Checks only what is objectively derivable from code/registry data: identification (resolved profile vs. the profile the fixture is filed under), mandatory-clause coverage against the registry's own `required_clauses`, dangerous-clause detection on fixtures deliberately authored with an embedded abusive addendum, recommendation-wiring completeness, and PDF generation. **Risk-score correctness is intentionally left as PENDING** — asserting it would mean inventing the legal ground truth that is Ilham's to supply, which is exactly the evidence-fabrication problem the review flagged.
* Audited the existing fixture corpus (`tests/fixtures/{pdf,docx,txt}`, 78 files) by actually running every candidate through the pipeline rather than trusting filenames. Result: **27 of the 33 target fixtures (11 profiles × 3) are confirmed correct and wired in.** The other 6 are genuine gaps, not oversights:
  * `lease_agreement`, `commercial_agreement`, `non_disclosure_agreement`, `partnership_agreement`: only 2/3 distinct fixtures exist or resolve correctly.
  * `purchase_agreement`: only 1/3.
* **4 real misclassification bugs surfaced by running the corpus** (not visible to smoke testing):
  1. `docx/02_employment_fr.docx` (French employment contract) → misclassifies as `memorandum_of_understanding`.
  2. `txt/13_medium_risk_lease_nl.txt` (Dutch lease) → misclassifies as `general_contract`.
  3. `docx/03_nda_nl.docx` (Dutch NDA) → misclassifies as `general_contract`.
  4. `txt/bench_purchase_agreement_neg.txt` (purchase agreement + injected abusive clause) → misclassifies as `telecommunications_agreement`. **This is the most concerning one**: injecting abusive/leonine text changes the *detected document type*, not just the risk score, indicating the L2 classifier is not robust to red-flag content.
* On the 27 fixtures that do run: 27/27 pass on all four assertable dimensions above. Full detail in [original11_corpus_report.json](file:///home/stardhoom/LDV/ldv-backend/tests/original11_corpus_report.json).
