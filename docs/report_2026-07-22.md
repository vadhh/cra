## 2. Summary of Changes Today

### 2a. Fix: orphaned `it_services_contract` label (`34d965c`)

`_keyword_doc_type()`'s hardcoded keyword-match chain returned the literal string `"it service agreement"` — a legacy label from a pre-consolidation per-file profile format (`detector/profiles/it_service_agreement.json`, confirmed not loaded by any live code path; `profile_registry.py` only reads `registry_v1.json`). That string never matched `it_services_contract`'s actual registry `display_name` (`"IT Services Contract"`), so a clean, unambiguous "IT services" text silently failed to resolve to any profile at all.

Fix: aligned all 3 occurrences (title-match branch, body-snippet branch, `_registry_keyword_fallback`'s `covered` set) to `"it services contract"`, matching the registry's own display name.

Verified: full suite unchanged; collision-pair #5 (`it_services_contract` vs `service_agreement`) now resolves correctly on the A-signal (A-signal baseline 7/13 → 8/13, zero other rows changed); original-11 corpus harness unchanged at 31/33.

### 2b. Fix: `outsourcing_agreement` → `employment_contract` misclassification (`7845851`)

Root cause distinct from 2a. `_keyword_doc_type()`'s header/title-matching pass (used to force an *unconditional* override of NLI, regardless of NLI's confidence, per the "ground truth title matched explicitly" rule) uses plain substring checks (`x in header`), unlike the body-snippet pass just below it, which already uses the word-boundary-aware `_has_word()` helper. The word `"employee"` substring-matches inside `"employees"`, so an outsourcing-agreement fixture that incidentally mentions "the transition of affected employees" false-triggered a title match on the *employment* branch — overriding NLI's actual top pick, which was correct and confident (`service agreement`, 0.93).

I first tried generalizing the fix to all 18 header-matching branches (switching every one from substring to `_has_word()`, matching the body-pass convention). That broke a different case: `"it service"` (singular) no longer word-boundary-matched against `"IT Services"` (plural) in the it_services_contract test text, regressing collision A-signal 8/13 → 7/13 and corpus recommendation-wiring 31/31 → 29/31. Reverted that broad attempt and scoped the fix to the single `employment` branch only, which was the actual source of this specific misfire.

Verified: full suite unchanged; original-11 corpus harness unchanged (31/33, 0 known bugs, 31/31 recommendation wiring); collision-pair #8 changed from resolving to an unrelated wrong profile (`employment_contract`) to resolving to the already-documented, already-triaged `service_agreement` collision (still reported as a FAIL by the test's strict pass criterion — it's the *expected* ambiguity per `classifier_collision_pairs.md`, not a new bug) — zero other rows changed.

### 2c. Decided and fixed: `maintenance_contract` → `employment_contract` (`2c51e7e`)

Earlier today this was flagged as needing a joint decision on override-logic strategy, since none of `classify_document_type()`'s 4 existing override conditions caught it (NLI confidently, wrongly, picked `employment_contract` — a *validated* profile — over a weak generic keyword match, with no title-match and no draft-vs-validated signal to intervene). On closer inspection this framing was too broad: it isn't an override-*logic* problem at all. The fixture's text titles itself `"MAINTENANCE CONTRACT."` verbatim — `maintenance_contract` was simply never added to `_keyword_doc_type()`'s hand-tuned title-matching chain, even though other draft-status profiles (`construction`, `insurance`) already have branches there. It's a gap in an existing, already-precedented pattern, not new scope.

Auditing that chain against the live registry (the same check that caught `it_services_contract` this morning) surfaced 2 more instances of the identical bug: `"construction agreement"` and `"insurance agreement"` don't match the registry's actual names (`"Construction Contract"` / `"Insurance Contract / Policy"`), so genuine construction/insurance contract text was silently unresolvable. Fixed all 3 together.

First attempt at the new `maintenance` branch used a bare substring check (matching the existing `construction`/`insurance` style) and immediately regressed collision-pair B-signal 11/13 → 7/13 — turned out test filler text across 4 *unrelated* pairs incidentally contains "maintenance agreement" as boilerplate wording (`"...perform maintenance agreement ... services..."`). Rescoped to the registry's own defined alias phrase (`"maintenance contract"`, not `"agreement"`), word-boundary matched. Re-verified clean.

Verified: full suite 108/108, original-11 corpus unchanged (33/33, 0 known bugs), collision-pair A-signal 8/13 → 9/13 with zero other rows changed, `construction_contract`/`insurance_contract` manually confirmed to now resolve correctly.

### 2d. Test: closed both fixture-content gaps (`54e0ea4`)

`commercial_agreement` and `partnership_agreement` were the 2 remaining profiles below 3/3 verified fixtures (§4 of `report_2026-07-21.md` treated this as a corpus-content gap for Ilham/Afridho, not code — no 3rd *distinct* fixture existed).
