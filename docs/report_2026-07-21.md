## 3. Summary of Changes Today (2026-07-21)

### 3a. Fix: 4th corpus misclassification bug (closes punch-list item 2)

`bench_purchase_agreement_neg.txt` was misclassifying as `telecommunications_agreement` instead of `purchase_agreement`. Root cause was different from the other 3 bugs fixed yesterday: keyword matching correctly found "purchase agreement" (hit_count=5), but NLI confidently (0.97) picked a wrong but *specific* (non-generic) draft-status label instead, and the override logic only trusted keyword evidence over NLI when NLI's pick was generic or low-confidence.

Before fixing, checked `worker._needs_confirmation()` and confirmed it already forces mandatory human confirmation whenever a classified profile's registry status isn't `validated` — `telecommunications_agreement` is `draft`. That meant the bug's real blast radius was a wrong *suggestion* already requiring human sign-off, not a silently-wrong final result, which justified a scoped fix: added a 4th override condition to `classify_document_type()` — when NLI's top pick resolves to a `draft`-status profile and a strong keyword match resolves to a `validated`-status profile, trust the keyword match. Deliberately narrow (can't second-guess NLI when both candidates are already `validated`), and reuses the existing `keyword_match` gate path — no `worker.py` changes needed.

Verified: full suite unchanged, full 78-fixture corpus rescan showed exactly one row change (the target fixture) and zero regressions. [run_original11_corpus.py](file:///home/stardhoom/LDV/ldv-backend/tests/run_original11_corpus.py) now reports **31/33 target fixtures, 0 known bugs**. Remaining 2 gaps (`commercial_agreement`, `partnership_agreement`) are corpus-content gaps — no 3rd distinct fixture exists — for Ilham/Afridho to fill, not code defects.
